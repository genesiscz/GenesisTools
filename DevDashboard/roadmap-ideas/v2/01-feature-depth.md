# Lens: The Power-User Depth Pass — daily-use screens & ambient polish

> A senior-product-designer pass that *deepens what already ships* (Pulse, ttyd terminals, Q&A, Obsidian, tmux/cmux, claude-usage, daemon, containers) into rich power-user screens — plus HomeKit ambient signals, a real automation rule-builder, and the Apple-ecosystem glance layer. Every idea names BOTH a web surface (multi-pane, keyboard-driven) and a mobile surface (Live Activity / widget / Watch / Shortcut). No moonshots — just delightful, shippable polish.

---

## Pulse (metrics)

- **Process Inspector with per-process history** `[feature]` — Today's `topProcesses` is a static RSS table. Make each row tappable → a detail panel that charts that PID's CPU + memory over the same 30m/2h/6h/24h ranges Pulse already keeps, plus parent/child tree, open files count, and a "this is new since 1h ago" badge. **Web:** a split view — process list left, live sparkline + kill/renice/`sample <pid>` actions right, sortable by Δ-over-window not just current. **Mobile:** long-press a process → context sheet with "watch this process" (fires a push if it crosses a memory/CPU band) and a Lock-screen Live Activity for a watched process during a long render. *Rationale: the #1 power-user follow-up to "CPU is at 90%" is "which process, and is it climbing" — the snapshot already has the PIDs, we just aren't keeping their history yet.*

## Terminals / ttyd

- **Multi-pane terminal grid + scrollback search** `[feature]` — Today ttyd is one interactive pane at a time. Add a tiling layout (2×2 / 1+2 / focus) that mounts several ttyd panes at once, each pinned to a named tmux/cmux session, with a per-pane title bar showing the live cwd + last exit code. A "⌘F across panes" searches all visible scrollbacks and jumps to hits. **Web:** resizable split grid with saved layouts ("frontend+backend+logs"), drag a session from the sidebar into a pane. **Mobile:** a horizontally-paged pane carousel (swipe between sessions) with a persistent bottom key-bar; pinch to see the 2×2 overview, tap to zoom one full-screen. *Rationale: power users live in 3-4 panes; a single-pane phone terminal is the biggest gap vs a desktop tmux setup.*

## Q&A stream

- **Agent-Waiting Triage Inbox** `[feature]` — Reframe the `qa-sse` feed from a chronological log into a triage queue: pull the `agent-signal` "waiting on input" events to the top as actionable cards (which session, the question text, time-blocked), with question/answer pairs grouped as threads and the existing read/`supersededBy` state driving an unread badge + "mark all read." **Web:** a three-column inbox (Needs You · Recent threads · Resolved) with `j/k` navigation and inline reply that writes straight into the originating tmux/cmux pane. **Mobile:** the unanswered-agent count becomes the app badge + a Dynamic Island "1 agent waiting 4m" pill; tap a card → quick-reply chips ("yes / no / proceed / explain") that send into the pane without opening the full terminal. *Rationale: the SSE stream already distinguishes question/answer/agent-signal — surfacing "who's blocked on me right now" is the daily job, and it's buried in a flat feed today.*

## Obsidian

- **Session Journal & one-tap capture** `[feature]` — Beyond read/publish, make the vault a *destination* for what the dashboard sees: a "Send to daily note" action that appends a timestamped block (a Q&A answer, a Pulse anomaly, a terminal selection, a finished daemon run) under `## Notes` of today's note, and an auto-built "Dev Log" note that links the sessions/runs you touched today. **Web:** a right-rail capture composer with a live preview of the daily-note section and backlinks to the source card. **Mobile:** a system Share-sheet target + a Shortcut ("Capture to today's note") and a Lock-screen widget showing today's unchecked tasks pulled from the vault. *Rationale: the vault browse/publish plumbing exists; the missing power-user loop is frictionless capture *from* the dashboard *into* notes, so context isn't lost.*

## tmux / cmux sessions

- **Session Board — the visual layout map** `[screen]` — A dedicated screen (not the terminal view) that renders every tmux/cmux session as a card showing its window/pane mini-map, the live last-line of each pane, cwd, attached/detached state, and idle time — driven by the cmux poller + tmux hub. Drag to reorder, group by project, rename inline, and "new session from template" (preset window/pane splits + startup commands). **Web:** a board/grid with hover-to-peek pane preview and bulk actions (kill idle, send `git status` to all). **Mobile:** a vertically-scrolling card list with a colored idle/active dot, swipe-to-kill, and a Home-screen widget showing the count of active vs idle sessions. *Rationale: power users juggle 8+ sessions; today you can only attach to one — a board that shows *all* of them at a glance is the orientation layer that's missing.*

## claude-usage

- **Usage Analytics screen — burn-rate, per-project, forecast** `[feature]` — Turn the `claude-usage` aggregator into a real analytics screen: tokens & cost over time (day/week/month), split by project/model, a "burn rate vs your plan window" gauge, top sessions by spend, and a simple forecast ("at this rate you'll hit the 5-hour cap in ~40m"). **Web:** stacked-area cost chart + a sortable per-project table with CSV export and a date-range picker. **Mobile:** a compact "today / this week" card, a Lock-screen widget with the burn-rate gauge, and a Live Activity that ticks during an active 5-hour window so you see the cap approaching. *Rationale: the aggregator already computes totals; power users want to *see the curve and not get rate-limited mid-flow* — that's a screen, not a number.*

## daemon

- **Run Timeline + log-tail with anomaly markers** `[feature]` — A timeline screen built on the daemon-view runs aggregator: each run as a horizontal bar (queued → running → done/failed) colored by status, with duration, exit code, and a sparkline of its own runs' historical durations so a slow/anomalous run stands out. Tap a run → tail its log with auto-highlighted error/warn lines and a "jump to first error." **Web:** a Gantt-style timeline with filter chips (failed only, > p95 duration) and a side log-tail pane. **Mobile:** a vertical run feed; toggle "watch this run" → completion/stall push + a Live Activity showing elapsed vs typical duration. *Rationale: the runs data exists but is shown as a list — the daily question is "is this run normal or stuck," which needs the historical-duration comparison and a live tail.*

## containers

- **Container Control Panel with live stat sparklines** `[feature]` — Promote the Docker view from a read list to a control panel: per-container CPU/mem/net/disk-IO sparklines (polled like Pulse), state (running/exited/restart-looping), health-check status, port map, and inline start/stop/restart/`logs -f` actions. A "restart loop" detector flags containers that have restarted N times. **Web:** a table-of-cards with expandable rows holding the live stats chart + a log-tail drawer and an "exec a shell" button that opens a ttyd pane into the container. **Mobile:** swipeable container cards with a status dot, tap-to-restart with confirm, and a widget showing how many containers are healthy vs down. *Rationale: containers already report stats; the power-user loop is "see it misbehaving → restart it / read its logs" without dropping to a laptop.*

## HomeKit (integration / ambient)

- **Ambient Dev-Status driver — lights & scenes react to your machine** `[integration]` — A HomeKit bridge that maps dashboard signals to home accessories: build fails → desk lamp flashes red; an agent is waiting on input → bulb pulses amber; CPU pegged / disk filling → a slow warning glow; all-green idle → calm cyan. A "Deep Work" scene can be triggered when a long build/agent run starts (dim lights, set a Focus). **Web:** a mapping screen — signal (left) → HomeKit accessory + color/scene (right) — with a "test flash" button per rule and a live "current ambient state" preview tile. **Mobile:** native HomeKit pairing via HomeKit framework + a Shortcut/Siri hook ("set dev status light"), and the same signal→light mapping editable on the phone. *Rationale: the user explicitly asked for smart-home ambient signals — turning the existing build/agent/metric events into a glanceable physical signal is the marquee delight feature, and the event sources already exist (qa-sse, daemon runs, Pulse thresholds).*

## Automation (rules engine)

- **Visual Rule-Builder — "when <signal> then <action>"** `[automation]` — A first-class automation screen: a list of rules each composed of a trigger (Pulse threshold, agent-waiting, daemon run done/failed, container down, claude burn-rate high), optional conditions (time window, Focus on/off, only-this-project), and one or more actions (push notify, run a saved terminal command, fire a webhook, trigger a HomeKit scene, toggle a Focus, start a Live Activity). Rules show a run history ("fired 3× today"). **Web:** a card-per-rule builder with dropdown trigger/condition/action rows, a dry-run/test button, and an "automation log" of every fire with its outcome. **Mobile:** a simplified rule list with enable/disable toggles, plus exporting any rule as an Apple Shortcut so it can be driven by Siri or the Shortcuts automations. *Rationale: this is the connective tissue the user asked for — it unifies alerts, HomeKit, commands, and Focus behind one "if this then that" surface instead of scattering toggles across screens.*

## Apple ecosystem (glance layer)

- **The Glance Stack — Live Activity, Dynamic Island, Watch & widgets** `[delight]` — A coherent ambient layer rather than one-off bolt-ons: a Live Activity + Dynamic Island that shows the highest-priority live state (agent waiting / build running / long job % / top metric) with compact + expanded layouts and one tap-to-act; a Home-screen widget gallery (Pulse dot, build status, active-session count, claude burn-rate, agents-waiting badge) the user composes; a Lock-screen circular complication; and an Apple Watch app for agent-waiting tap-to-approve + Pulse glance + alert triage from the wrist. **Web:** a "Glance Designer" screen to choose which signal each widget/complication shows and preview them (the same config drives the phone). **Mobile:** the actual WidgetKit / ActivityKit / WatchKit surfaces, configured from the in-app Glance settings. *Rationale: the user named Live Activities, Dynamic Island, Watch, and widgets specifically — bundling them into one composable "what do you want to glance at" system makes them feel designed, not stapled on.*

## Cross-cutting power-user screen

- **Command Palette + Search-Everything + Saved Views** `[screen]` — One ⌘K palette that fuzzy-searches *across every feature at once*: sessions, processes, Q&A threads, vault notes, daemon runs, containers, and actions ("restart nginx", "open backend pane", "publish note") — results grouped by source with a unified recents list. Pin any configuration of panes/cards as a named **Saved View** ("Morning standup": Pulse + agents-waiting + last night's runs) launchable from the palette or a home tile. **Web:** the palette is keyboard-first (`⌘K`, arrow-nav, `Enter` to act); Saved Views render as a customizable multi-pane dashboard grid. **Mobile:** a pull-down search field on every screen + Saved Views as swipeable dashboard pages and a Shortcut to jump straight to a named view. *Rationale: with 8+ feature areas the navigation cost compounds; a single search/command surface + saved layouts is the power-user accelerator that ties the whole app together.*
